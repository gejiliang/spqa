## 第一阶段：起点——AI-Native Organization 的构想

**GG 提出核心想法：**
想打造一个 AI-native organization，基于 OpenClaw，但不全是由 agent 组成，而是把自己（人类）也加入这个组织，协同工作。

**Claude 调研了 OpenClaw 现状：**
- OpenClaw 是一个开源 AI agent 框架，由 Peter Steinberger 创建，2026 年初爆火（GitHub 270K+ stars）
- 本质是一个 AI Agent 执行框架，连接 AI 模型到日常通讯工具（WhatsApp、Telegram、Discord、Slack 等）
- 支持持久记忆、cron 定时任务、webhook 触发器、skill 插件系统
- 已有用户搭建多 agent 架构（如 solo founder 用 4 个 agent 各司其职）
- 安全方面存在 prompt injection 风险，需注意权限边界

**Claude 同时调研了 AI-native 组织的行业趋势：**
- McKinsey 描述的 agentic organization：运营模式从功能性孤岛演变为扁平的人机协作团队网络
- 人类通常在 loop 之上进行方向把控，选择性在关键节点进入 loop 内部
- 核心问题：角色定义、共享状态/上下文、通讯拓扑、治理和安全

## 第二阶段：初版工作室架构设计

**GG 的选择：**
- 组织方向：综合型个人工作室（多种业务）
- 自身角色：CEO/决策者——把控方向，审批关键节点
- 初始规模：8+ 个 agent 角色（完整组织模拟）

**Claude 设计了初版架构（四层结构）：**

**CEO（你）→ 高管层 → 管理层 → 执行层 + 治理层**

各角色：
1. **参谋长（Chief of Staff）** — 直接助手，协调所有 agent，汇总信息，管理日程
2. **项目经理（Project Manager）** — 跟踪项目进度，分配任务，确保交付
3. **创意总监（Creative Director）** — 内容创作、设计方向、品牌调性
4. **研究主管（Research Lead）** — 深度研究、信息收集、行业分析
5. **运营执行官（Operations）** — 自动化流程、系统维护、技术基础设施
6. **财务分析师（Finance）** — 收支追踪、预算管理、投资分析
7. **对外沟通官（Communications）** — 社交媒体、邮件管理、社群运营
8. **质量守门员（QA/Critic）** — 审查产出质量、一致性、安全性
9. **知识管理员（Archivist）** — 组织记忆、知识库维护、经验沉淀

**设计了六种沟通协议：** 晨报上行、任务下达、横向协作、审批上报、质量巡检、周度复盘

**设计了四级自主权限：** 全自主 → 有限自主 → 协作 → 需审批。核心原则：默认给予最大自主权，逐步收紧比逐步放开更高效。

## 第三阶段：引入三省六部制对比

**GG 的请求：** 把三省六部的组织架构、沟通协议和自主权限也画出来，要对比进行修改。

**Claude 的误解：** 最初以为 GG 指的是历史上的隋唐三省六部制，画了一版历史制度的架构图。

**GG 纠正：** 指的是 GitHub 上的 Edict 项目（github.com/cft0808/edict）——一个基于 OpenClaw 的多 Agent 编排系统，用三省六部制作为架构隐喻。

**Claude 重新调研了 Edict 项目的实际架构：**

Edict 的核心设计：
- 9 个 AI Agent 组成三省六部
- **中书省** — 接旨、规划、拆解子任务
- **门下省** — 审议方案、准奏/封驳（核心：制度性审核）
- **尚书省** — 派发任务、协调六部、汇总回奏
- **六部**（户礼兵刑工）— 专项执行
- **早朝官** — 情报采集和推送

Edict vs 其他框架的核心差异：制度性审核 + 完全可观测 + 实时可干预

关键约束——严格的权限矩阵：
- 六部只能向尚书省回报，不能横向沟通，不能越级上报
- 三省之间可互相通信
- 皇上通过看板观测全局，通过消息渠道只和中书省对话

## 第四阶段：核心架构辩论——三省 vs 元老院

**GG 提出关键问题：** 三省把决策审核调度派发独立，和元老院-凯撒的结构，哪种更 AI-native？

GG 描述的元老院模式：针对每个项目，若干个 agents 进行讨论，通过投票或类似陪审团全票通过，再交由后续执行层。

**Claude 的分析——元老院更 AI-native，原因有三：**

1. **LLM 的失败模式是单点过度自信。** 三省模式假设流程对了结果就对，但如果门下省和中书省用同一模型，封驳质量存疑。元老院通过多个独立视角并行，天然对冲了 LLM 的推理偏差——类似 ensemble learning。

2. **AI agent 没有人类的注意力瓶颈。** 三省六部诞生于人类精力有限的背景，但 agent 可以同时启动多个并行思考，成本只是多几次 API 调用。三省的串行流水线浪费了 AI 最大的优势：并行性。

3. **元老院天然支持异构。** 不同元老可配不同模型或完全不同的 system prompt 视角（风险厌恶型 vs 激进型，技术视角 vs 商业视角），在三省模式里很难实现。

**但三省的优势：** 执行层的调度。元老院适合决策，不适合执行编排。

**达成共识：决策层用元老院，执行层用尚书省+六部模式。**

## 第五阶段：元老院机制深化

### 5.1 秘书角色的设计

**GG 的构想：**
- 元老不需要持久化，可以按领域、分工划分，每次从元老池里选出适合当前项目的若干个
- 新增秘书角色，处于凯撒和元老院之间
- 秘书平时是生活助理，同时有个 subagent 专门做元老挑选

**Claude 的分析：**
- 秘书是「两个功能叠加在一个持久化 agent 上」
- 秘书是整个系统里唯一同时理解「你的个人上下文」和「元老池能力图谱」的角色
- 元老的非持久化很合理——跟 serverless function 理念一致
- 关于筛选机制：建议让 subagent 先对议题做快速分析（涉及哪些领域、需要什么视角），再基于分析匹配

### 5.2 元老院的讨论机制

**GG 提出了一套完整的议制模型：**

核心挑战：如何避免「复读机效应」或无意义循环。

GG 提出四种经典方法：
1. **德尔菲法** — 背对背匿名迭代，避免强势模型带偏节奏
2. **六顶思考帽** — 强制性视角切换，防止方案过于平庸
3. **红蓝军对抗** — 防御性共识，挤掉方案水分
4. **罗伯特议事规则** — 程序化共识，动议+辩论+投票

**GG 的推荐混合架构——「总分总」模式：**

| 阶段 | 机制 | 动作 |
|------|------|------|
| 发散期 | 六顶思考帽 | 不同侧重点的 AI 各自输出专业意见 |
| 冲突期 | 红蓝对抗 | AI 之间互相 Review，指出逻辑矛盾 |
| 收敛期 | 德尔菲法 | 协调员提取公约数，1-2 轮匿名投票修正 |
| 产出期 | 蓝帽总结 | 输出剔除冲突、保留多维视角的统一方案 |

**GG 的关键避坑指南：** AI 讨论中最容易出现「老好人」现象——A 说了一点，B 说「我同意 A，但补充…」导致方案越来越冗长。建议明确：「若无新增价值或实质性反对理由，禁止表达附和。」

**Claude 高度认同这套设计，** 补充：这条「禁止无实质附和」的规则对 LLM 太重要了，因为 LLM 天然倾向于礼貌性同意。

## 第六阶段：执行层和持久化策略讨论

### 6.1 元老院决议包含执行方案

**关键共识：** 元老院的产出不只是抽象决策，而是可执行方案——包括需要哪些执行角色、怎么分工、交付标准。这在四阶段议制的收敛期和产出期已经具体化了。

**由此推导：** 三省里的「尚书省」可以大幅简化甚至取消，因为任务拆解和角色分配元老院已做，剩下的只是「按方案实例化执行 agent」这个技术动作，可以是秘书的一部分职能。

### 6.2 执行层的划分方式

GG 在两个方向之间摇摆：
- 偏业务向（创意/研究/运营/财务/沟通）
- 混合（核心持久化 + 其他动态）

**Claude 的建议——用一个简单标准判断：这个角色需不需要跨项目的持续记忆？**
- 需要记忆 → 持久化（秘书、知识管理员、财务）
- 不需要记忆 → 动态化（临时执行 agent，由知识管理员归档成果）

### 6.3 持久化需求分两类

**GG 补充了重要信息：** 持久化不只是综合办公室，还有些项目（如量化交易团队）需要全员持久化。

**最终确定三种 Agent 生命周期：**
1. **Always-on** — 组织基础设施，24/7 运行（秘书、知识管理员、财务、网管、情报、社交分析）
2. **Project-persistent** — 特定项目全员常驻（如量化团队）
3. **Ephemeral** — 按需实例化，任务完成即销毁（元老 + 临时执行 agent）

### 6.4 持久化项目组的决策机制

**GG 的选择：** 持久化项目组由凯撒直接指挥，不经过元老院。原因：量化这种实时性强的项目，多一层元老院讨论是延迟。

## 第八阶段：禁卫军通用化

**GG 的要求：** 把量化交易军团改成类似特勤组的罗马背景团队，不涉及具体业务，方便后续设计。

**Claude 的设计：**
- 禁卫军（Praetorian Guard）是一个分类
- 下辖多个 **Cohors Specialis（特勤队）**
- 凯撒按需组建、命名、编号（CS-α、CS-β…）
- 标准角色骨架（可按业务增减）：
  - Strategus · 谋略官 — 策略制定 · 方向把控
  - Custos · 守卫官 — 风控管理 · 边界守护
  - Executor · 执行官 — 核心任务执行
  - Tabularius · 记录官 — 数据维护 · 状态追踪

**禁卫军条令（Lex Praetoriana）：**
1. 每队全员持久化，项目存续期内 7×24 运行
2. 凯撒直接指挥，不经执政官和元老院
3. 可根据业务需要增减队内角色
4. 成果由史官归档，定期向凯撒汇报

## 第九阶段：可读性优化

**GG 的反馈：** 拉丁文名称太深奥，每个角色都要加上易懂的中文名称或职能说明。
