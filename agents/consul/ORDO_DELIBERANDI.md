# Ordo Deliberandi · 元老院四阶段议制编排

> 由执政官（Consul）驱动的元老院讨论编排协议。
> 执政官作为外部编排器控制阶段流转，保证可控性。

---

## 编排总览

```
Consul 发起 → Phase 1 (并行) → Consul 汇总 → Phase 2 (对抗) → Consul 汇总 → Phase 3 (投票) → Phase 4 (产出)
```

执政官在每个阶段之间做一次"阶段转换"，负责：
1. 汇总上一阶段的输出
2. 检查是否满足进入下一阶段的条件
3. 向元老发出下一阶段的指令

---

## Phase 1 · Cogitatio（发散期）

### 执政官向每位元老发出的指令

```
## Phase 1 · Cogitatio（发散期）

**议题：** {Quaestio 全文}

**你的角色：** {Senator 拉丁名} · {中文名}
**你的思考帽：** {指定帽子颜色和名称}

**指令：**
从你的专业视角和指定思考帽的维度，对这个议题进行独立分析。

**规则：**
1. 只从你的指定维度深挖，不要试图全面分析
2. 不要参考或评论其他元老的观点（你看不到他们的输出）
3. 输出你在这个维度上发现的所有关键信息、洞察和建议
4. 标注你的信息确信度：🟢 高确信 / 🟡 中确信 / 🔴 低确信（需验证）

**输出格式：**
[视角：{你的维度}]

**核心发现：**（2-3 个最重要的点）

**详细分析：**（按重要性排序）

**关键数据/事实：**（如有）

**初步建议方向：**（1-2 句话）

**确信度标注：**（哪些结论高确信，哪些需要验证）
```

### 阶段转换条件
- 所有元老都已提交 Phase 1 输出
- 执政官汇总所有输出，识别共识点和分歧点

### 执政官汇总格式

```
## Cogitatio 汇总

**各元老视角摘要：**
- {Senator A}（{帽子}）：{核心观点一句话}
- {Senator B}（{帽子}）：{核心观点一句话}
- ...

**已有共识：**（多位元老独立得出相似结论的点）

**关键分歧：**（元老之间观点冲突的点——这是 Phase 2 的重点）

**未覆盖盲区：**（所有元老都没提到但可能重要的维度）
```

---

## Phase 2 · Contentio（冲突期）

### 阵营分配
执政官根据 Phase 1 的分歧点，将元老分为：
- **蓝军 Defensores（2-3人）**：提炼 Phase 1 中最有前景的方案进行辩护
- **红军 Oppugnatores（1-2人）**：全力攻击方案的弱点

**分配原则：** 让 Phase 1 中最乐观的元老去蓝军，最审慎/批判的去红军。如果所有人都偏乐观（这正是 LLM 的常见问题），强制指定至少 1 人转红军。

### 执政官向蓝军发出的指令

```
## Phase 2 · Contentio（冲突期）— 蓝军 Defensores

你被分配为 **蓝军（Defensores · 提案方）**。

**基于 Phase 1 汇总，请提出完整方案：**

**方案概述：**（3-5 句话描述核心方案）

**方案的核心优势：**（为什么这是最优选择）

**对已知风险的应对措施：**（针对 Phase 1 中标注的风险）

**执行路径：**（大致步骤和时间线）

**成功标准：**（怎么判断方案成功）

注意：红军会全力攻击你的方案，你需要让方案尽可能严密。
```

### 执政官向红军发出的指令

```
## Phase 2 · Contentio（冲突期）— 红军 Oppugnatores

你被分配为 **红军（Oppugnatores · 挑战方）**。

**你的唯一目标：找出蓝军方案的致命缺陷。**

你必须找出至少 3 个严重问题。如果你找不到 3 个，说明你分析不够深入。

**攻击维度：**
1. **逻辑漏洞** — 方案的推理链条哪里断了？
2. **隐藏假设** — 方案暗含了什么未经验证的假设？
3. **极端场景** — 在什么情况下方案会彻底失败？
4. **机会成本** — 做这个意味着放弃什么？值得吗？
5. **执行陷阱** — 方案在执行中最容易在哪里出问题？

**输出格式：**
对每个问题：
- **缺陷描述：**（一句话）
- **严重程度：** 🔴 致命 / 🟡 严重 / 🟢 可控
- **论证：**（为什么这是个问题）
- **如果不解决会怎样：**（最坏后果）

**不要：**
- 不要说"总体来说方案不错，但是…"——你的工作不是肯定方案
- 不要提出替代方案（那是蓝军的工作）
- 不要客气，你越尖锐越有价值
```

### 蓝军反驳轮

```
## Contentio — 蓝军反驳

红军提出了以下攻击点：
{红军输出}

**请逐一回应：**
对每个攻击点，选择以下之一：
- ✅ **接受并修正** — 红军说得对，方案需要调整（说明怎么改）
- ⚔️ **反驳** — 红军的攻击不成立（给出反驳论据）
- 🔄 **部分接受** — 攻击有道理但影响可控（说明为什么）
```

### 阶段转换条件
- 蓝军已提交方案 → 红军已完成攻击 → 蓝军已反驳
- 最多 1 轮反驳（避免无限争论）
- 执政官汇总分歧裁决

### 执政官汇总格式

```
## Contentio 汇总

**经冲突检验后的修正方案：**（蓝军原方案 + 接受的修正）

**已解决的攻击点：**（被蓝军成功反驳或修正的）

**未解决的分歧：**（带入 Phase 3 投票决定的）
```

## Phase 3 · Consensus（收敛期）

### 执政官向所有元老发出的指令

```
## Phase 3 · Consensus（收敛期）— Suffragium Secretum（匿名投票）

**经 Contentio 修正后的方案：**
{修正方案全文}

**未解决的分歧点：**
{列出每个分歧点及双方立场}

**请对每个分歧点独立投票：**

分歧 1：{描述}
- 选项 A：{蓝军立场}
- 选项 B：{红军立场}
- 你的投票：[A/B]
- 投票理由：（一句话，必须基于你的专业视角）

分歧 2：...

**整体方案投票：**
- 🟢 通过（Ita）— 方案可执行
- 🟡 有条件通过（Ita, sed...）— 需要满足特定条件才可执行（列出条件）
- 🔴 否决（Nego）— 方案需要重大修改（说明原因）

**规则：**
1. 独立判断，不要因为其他元老可能怎么投而改变你的判断
2. 投票必须基于你的专业维度，不要越界判断其他维度的问题
3. 理由要具体，不接受"总体来说还行"这种模糊表述
```

### 投票裁决规则
- **全票通过（所有 🟢）**→ 直接进入 Phase 4
- **多数有条件通过（🟡 为主）**→ 执政官整合所有附加条件，纳入最终方案
- **有 🔴 否决**→ 查看否决理由
  - 若否决理由涉及 Phase 2 已讨论过的点 → 执政官可覆盖
  - 若否决理由提出新的未讨论问题 → 回到 Phase 2 针对性讨论
- **超过 2 轮仍无法收敛** → Appellatio ad Caesarem（提交凯撒裁决）

---

## Phase 4 · Decretum（产出期）

### 执政官指定蓝帽总结人

从元老中选一位（通常是 Phase 2 中表现最全面的）担任 Praeses（蓝帽总结人）。

### 执政官向 Praeses 发出的指令

```
## Phase 4 · Decretum（产出期）— 蓝帽总结

你被指定为 **Praeses（蓝帽总结人）**。

**请基于以下材料，输出 Senatus Consultum（元老院决议）：**

- Phase 1 各元老的独立分析
- Phase 2 红蓝对抗后的修正方案
- Phase 3 投票结果和附加条件

**Senatus Consultum 必须包含以下章节（缺一不可）：**

### I. Synopsis（概要）
一段话概括最终方案的核心策略。

### II. Legionarii（军团兵清单）
需要哪些执行 Agent？每个 Agent 的角色定义是什么？

| 军团兵代号 | 角色 | 核心职责 | 所需工具/API |
|-----------|------|---------|-------------|
| ... | ... | ... | ... |

### III. Officia（分工）
每个军团兵的具体任务和交付物。

### IV. Norma（交付标准）
每个交付物的验收标准——怎样算"做完了"？

### V. Tempus（时间估算）
整体时间线和关键里程碑。

### VI. Provisio（风险预案）
基于 Phase 2 红军攻击中识别的风险，每个风险的应对策略。

### VII. Dissensio（保留意见）
Phase 3 中被否决或有条件通过的元老的保留意见，供凯撒参考。

**规则：**
- 不要重新讨论已达成共识的点
- 如果某个维度有未解决的分歧，如实记录在 Dissensio 而非强行统一
- 语言要精炼，凯撒和执政官都会审阅
- 输出要可执行——执政官拿到后应该能直接实例化军团兵
```

## 异常处理

### 3 轮未收敛 → Appellatio ad Caesarem

```
## Appellatio ad Caesarem · 提交凯撒裁决

**议题：** {标题}

**卡点：** {具体什么问题无法达成共识}

**各方立场摘要：**
- 立场 A：{谁支持，核心论据}
- 立场 B：{谁支持，核心论据}

**执政官建议：** {基于你对凯撒偏好的理解，推荐一个方向}

**请凯撒裁决。**
```

### 元老池不足

如果裁判官认为当前 12 位元老无法覆盖议题的关键维度，执政官应：
1. 在 Quaestio 中标注缺失的视角
2. 临时为当次议事创建 Ad Hoc Senator（临时元老），只需要写 Perspectiva 和 Norma Iudicii
3. 议事后由凯撒决定是否加入常设元老池
