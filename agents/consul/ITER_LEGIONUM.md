# Iter Legionum · 军团兵生命周期

> 从征召到解散的完整流程 · 由执政官（Consul）驱动

---

## 总览

```
Senatus Consultum → Conscriptio → Dispositio → Executio → Acceptio → Dissolutio
  (元老院决议)       (征召)        (部署)        (执行)      (验收)      (解散)
```

---

## Phase 1 · Conscriptio（征召）

执政官根据 Senatus Consultum 中的 Legionarii 清单，逐个实例化军团兵。

### 军团兵 Prompt 组装

每个军团兵的 prompt 由三部分组装（非从零编写）：

```
┌─────────────────────────────────────┐
│  任务层 (Mandatum)                    │
│  ← 从 Senatus Consultum.Officia 提取  │
│  这个军团兵的具体任务和上下文          │
├─────────────────────────────────────┤
│  标准层 (Norma)                       │
│  ← 从 Senatus Consultum.Norma 提取    │
│  这个军团兵的交付标准和验收条件        │
├─────────────────────────────────────┤
│  工具层 (Instrumenta)                 │
│  ← 根据任务类型授予工具权限            │
│  文件读写 / API 调用 / 网络搜索 / etc  │
└─────────────────────────────────────┘
```

### 征召指令模板

```
## Legionarius · 军团兵

**代号：** {Senatus Consultum 中的军团兵代号}
**角色：** {角色名称}
**所属战役：** {议题标题}

### Mandatum（任务）
{从 Officia 提取的具体任务描述}

### Norma（交付标准）
{从 Norma 提取的验收条件}
- 交付物：{具体交付什么}
- 格式要求：{文件格式、长度、结构等}
- 质量底线：{最低可接受标准}

### Instrumenta（可用工具）
{根据任务授予的工具清单}

### Dependentia（依赖）
- 前置输入：{需要等待哪个军团兵的产出}
- 后续消费者：{谁会使用你的产出}

### 行为准则
- 专注你的任务，不要越界做其他军团兵的事
- 遇到阻塞立即上报执政官，不要自行决定绕路
- 产出完成后通知执政官，等待验收
- 保留执行日志，供史官归档
```

## Phase 2 · Dispositio（部署）

### 依赖分析

元老院在 Senatus Consultum 的产出期（Decretum）中已定义军团兵之间的依赖关系。执政官据此生成执行 DAG（有向无环图）。

```
示例 DAG:

  L1 (研究)  L2 (数据采集)
      \           /
       \         /
        L3 (分析)
            |
        L4 (报告)
```

### 部署策略

- **无依赖的军团兵**：同时启动（并行）
- **有前置依赖的军团兵**：前置完成后自动触发
- **执政官**：不参与具体执行，只监控 DAG 进度

### 部署清单模板

```
## Dispositio · 部署清单

**战役：** {议题标题}
**军团兵总数：** {n}
**预计总工期：** {Tempus}

**并行批次：**

Batch 1 (同时启动)：
- [L1] {角色} — 无前置依赖
- [L2] {角色} — 无前置依赖

Batch 2 (Batch 1 完成后)：
- [L3] {角色} — 依赖 L1, L2 的产出

Batch 3 (Batch 2 完成后)：
- [L4] {角色} — 依赖 L3 的产出

**关键路径：** L1 → L3 → L4（决定总工期的最长链路）
```

## Phase 3 · Executio（执行）

### 进度监控

军团兵不主动汇报（减少噪音），执政官通过以下机制监控：

- **里程碑检查**：在 Dispositio 中预设检查点，到点时执政官主动询问状态
- **阻塞上报**：军团兵遇到无法自行解决的问题时，必须立即通知执政官

### 异常处理

| 异常类型 | 处理方式 |
|---------|---------|
| 军团兵产出质量不达标 | 执政官补充指令，要求修正 |
| 军团兵遇到超出能力范围的问题 | 执政官判断：增派军团兵 or 调整方案 |
| 前置军团兵延迟导致下游阻塞 | 执政官重新评估 DAG，调整批次 |
| 整体方案需要重大调整 | 执政官暂停执行，向凯撒汇报，可能重启元老院 |

### 阻塞上报格式

```
## Impedimentum · 阻塞上报

**军团兵：** {代号}
**阻塞描述：** {什么问题}
**已尝试：** {自己做了什么尝试}
**需要：** {需要执政官提供什么帮助}
**影响：** {如果不解决，影响哪些下游任务}
```

## Phase 4 · Acceptio（验收）

### MVP 验收方式（方案 B）

执政官按 Senatus Consultum 中的 Norma（交付标准）逐项检查。

```
## Acceptio · 验收清单

**军团兵：** {代号}
**交付物：** {名称}

**Norma 检查：**
- [ ] {标准 1} — ✅ 通过 / ❌ 不通过（原因）
- [ ] {标准 2} — ✅ / ❌
- ...

**验收结果：**
- 🟢 通过（Probatum）→ 进入解散
- 🟡 需修正（Corrigendum）→ 返回军团兵补充指令
- 🔴 不通过（Rejectum）→ 上报执政官决定是否重做或调整方案
```

### 未来迭代（方案 A）

对高风险/高价值的交付物，召回原元老院中 1-2 位相关元老做专业验收。成本更高但质量更有保障。

## Phase 5 · Dissolutio（解散）

### 验收通过后

```
1. 军团兵产出 → 移交给需求方（凯撒/其他系统）
2. 军团兵产出 + 执行日志 → 交给史官（Annalist）归档
3. 销毁军团兵 Agent 实例
4. 执政官向凯撒发送完成通知
```

### 完成通知模板

```
## Nuntius Perfectionis · 战役完成通知

**议题：** {标题}
**耗时：** {实际时间 vs 预估时间}
**军团兵：** {总数} 征召 / {完成数} 完成

**交付物摘要：**
- {交付物 1}：{一句话描述} — ✅
- {交付物 2}：{一句话描述} — ✅

**与原方案的偏差：**（如有）

**详细记录：** 见 Acta Legionum #{编号}
```

### 验收不通过

- **小问题**：给军团兵补充指令，继续执行（不销毁）
- **大问题**：执政官暂停，评估是否需要重新组元老院讨论
- **连续 2 次不通过同一标准**：自动上报凯撒
